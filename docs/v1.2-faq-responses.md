## CATEGORY 1 — Skepticism / "yeah right"

### Q: "How are you going to compete with Square / Toast / Clover?"

```
Honestly? I'm not trying to compete with them on features. I'm
competing on price and lock-in.

Square charges $89/month + 2.6%. Toast charges $69-165/mo + 2.5%.
OSPOS charges $0/month + 1% on top of Stripe's standard rate.

For a coffee shop doing $30k/mo in cards, that's $4k+/year I save them.
Most small merchants don't need 80% of what Toast ships. They need
to take payments without giving up 4%.

I'm one person. I'll lose to Square on features. I'll win on price
and on the fact that the merchant owns their Stripe account directly
and can fork the code if I disappear.
```

### Q: "What happens when you give up / disappear?"

```
That's exactly why it's open source under AGPL. Three things hold:

1. Your Stripe account is yours — Connect Standard, not a sub-account
   I control. Money flows merchant → Stripe → your bank. I never touch it.
2. Your menu and transaction data live on your phone, exportable as
   CSV anytime.
3. The server is small (~3k lines) and you can self-host it from the
   repo. Or fork it. Or pay anyone to keep it running.

If I get hit by a bus tomorrow, every merchant keeps running.
```

### Q: "How can you make money on 1%?"

```
Marginal — you're right. One merchant doing $100k/year in cards = ~$1000
of revenue for me. To live I need ~50 merchants. To be comfortable, 500.
To buy ramen for the kids, 5000.

I have 3 paying merchants today. We'll see.

The math works only if I keep the team at exactly one person. Which is
the point of the architecture (lean stack, no ORM, integer cents
everywhere) and the license (no SaaS competitor can fork-and-undercut).
```

### Q: "What's the catch?"

```
No catch. Three real constraints:

1. US-only at launch (Stripe's Tap to Pay rollout limit)
2. iPhone-only (Tap to Pay isn't on iPad yet — Apple's call, not mine)
3. The 1% is on top of Stripe's standard rate, not in place of it.
   So your total cost on a card transaction is ~3.7% + 5¢.

If those don't work for your business, that's fair.
```

## CATEGORY 2 — Architecture / technical

### Q: "Why React Native?"

```
Single-person team. I needed to ship iOS + (eventually) Android from
one codebase. React Native + Expo SDK 55 made that tractable. The
performance trade is fine for a register app — UI doesn't need to be
60fps butter, it needs to render a 3-column tile grid and tally
integer math correctly.

If I had a team of 5 I'd probably split into native iOS Swift +
native Android Kotlin. With one person, the RN bet paid off.
```

### Q: "Why no ORM? Why raw SQL?"

```
SQL is the data layer. ORMs add cognitive overhead, lock you to
their query planner, and obscure performance characteristics. For a
~3k line server with ~20 endpoints, raw parameterized queries are
faster to write, faster to debug, faster to optimize.

Same for the client SQLite layer. expo-sqlite gives you a clean
parameterized API; wrapping it in Drizzle or Prisma would have
saved maybe 100 lines and cost a week of "why doesn't this query
do what I want."
```

### Q: "Why AGPL specifically and not MIT/Apache/BSD?"

```
Payment processing has a SaaS-shaped competitive moat. Under MIT,
Toast could fork OSPOS, host it, charge merchants $99/mo, and never
contribute back. AGPL closes that loophole: if you run a modified
version as a service, you have to publish your modifications.

For a single-founder business competing on 1% vs incumbents at 2.5%,
AGPL is the only license that doesn't self-destruct economically.

Tradeoff: enterprise teams sometimes blacklist AGPL. Bitwarden,
Plausible, Mastodon, Ghost all made the same call for the same reason.
Fine company.
```

### Q: "How does the offline-first sync work?"

```
Cash transactions: 100% local SQLite. Never sync. Never leave the
phone. That's the privacy promise.

Card transactions: also local SQLite, but enter a sync_queue table.
A background process every 30s drains the queue to /sync/push on
the server. Server has its own synced_orders table for dispute
evidence + CSV export.

Honest note: the sync_queue insertion code has a gap right now —
queue table exists but no callsite writes to it. On the v1.1.x
punch list. Doesn't affect day-of merchants (orders work locally,
receipts work, Stripe receives the PaymentIntent directly).
```

### Q: "What's in the audit?"

```
External security audit handed off June 17 to a contributor focused
on iOS mobile security. Scope in repo at
docs/audit-plan-v1.1-to-v1.2.md.

Priority areas: menu import (accepts arbitrary JSON from user-picked
files), auth/JWT flow, Stripe Connect OAuth state handling, receipt
template (HTML rendered into emails). Findings land before any
v1.1.2 release.
```

## CATEGORY 3 — Stripe / payments / regulation

### Q: "Why Stripe and not Adyen / Braintree / Square / etc?"

```
Stripe Connect Standard is the only major payment platform that lets
merchants own their account directly (vs being a sub-account of mine).
That's the no-vendor-lock-in promise. Merchant signs up directly with
Stripe; I just broker the OAuth.

Tap to Pay on iPhone is implemented via the Stripe Terminal SDK —
same SDK Square Reader and other PSPs use under the hood. Stripe
won that integration first; everyone else followed.

If Adyen or Braintree ever match Stripe Connect Standard + Tap to
Pay on iPhone, I'd consider supporting both. Today, Stripe is the
only path.
```

### Q: "Is OSPOS PCI compliant?"

```
OSPOS itself doesn't touch card numbers. The Stripe Terminal SDK
captures the contactless tap and tokenizes the card on-device; only
the token ever reaches my server.

That puts OSPOS in PCI SAQ A scope (the simplest tier), which is
what you'd be in if you used Stripe directly via their SDK. The
hosted version's server is in scope for the bits that handle the
Stripe Connect token + the PaymentIntent metadata — none of which
includes raw card data.

Stripe's compliance documentation covers the full chain:
https://stripe.com/guides/pci-compliance
```

### Q: "What about EMV / chargebacks / disputes?"

```
Tap to Pay on iPhone is EMV-certified by Apple + the card networks.
Card brand fraud liability shifts to the issuer for contactless
EMV transactions, which is good for merchants.

For disputes: OSPOS has an in-app dispute evidence flow. When Stripe
sends a webhook saying "this transaction is disputed," the merchant
gets a notification, uploads receipts + product photos + customer
correspondence directly from their phone, and OSPOS submits the
evidence package to Stripe.

The whole flow is in src/screens/DisputesScreen.tsx if you want
to audit.
```

### Q: "Why no support for ACH / bank transfers / international cards?"

```
ACH: requires manual entry of routing + account, which is friction
the merchant has to deliver to the customer at checkout. Bad fit for
a tap-and-go POS. If it surfaces as demand I'll consider it.

International cards: Stripe Tap to Pay on iPhone supports them as
long as the merchant is in a supported country (US currently for
OSPOS; CA/UK/AU/NZ/SG/IE next). The CUSTOMER's card can be from
anywhere — same as any contactless tap.
```

## CATEGORY 4 — Roadmap / "when are you going to..."

### Q: "When Android?"

```
On the roadmap, not imminent. Tap to Pay on Android via Stripe
Terminal SDK is GA as of late 2024 in the US/CA/UK/AU/NZ/SG. The
work is real (Stripe Terminal Android SDK is different from iOS,
different test surface, different EAS build pipeline) — 6-8 weeks
of focused work. Probably v1.4 or v2 territory in 2026.

Honest: I'd rather get iOS to 100 merchants before splitting attention.
```

### Q: "When iPad?"

```
Tap to Pay on iPhone is iPhone-only. Apple hasn't extended it to
iPad. So a standalone iPad POS would require a separate Bluetooth
card reader, which defeats the "no hardware" pitch.

What might work: iPad as the counter UI + your iPhone (in a holster
on the counter) as the paired Tap to Pay reader. That's a v1.3
roadmap item. The architecture is sketched in
project_dual_device_killer_idea.md if you want a peek.
```

### Q: "When restaurant features (tables, kitchen tickets)?"

```
v2 territory. Probably Q4 2026 or early 2027. The architecture
already supports it (modifier groups + orders + soft deletes) but
the UX for table management, course timing, and kitchen ticket
routing is its own product. Want to win counter-service first.

If you operate a restaurant and would test a v2 alpha, email
phil@tttships.co.
```

### Q: "When grocery / barcode scanning?"

```
v3 territory. SKU-light retail (50-200 items) works today via the
tile grid. Real grocery (1000+ SKUs) needs camera-based barcode
scan + variable-price (weighed) items, neither of which are built.

If you run a corner store or specialty market, the existing v1.1
might be enough — 80% of small retail menus fit in tile-grid
browsing. Try the App Store version.
```

### Q: "When Europe / international?"

```
US-only at v1.1 launch. v1.2+ adds the Stripe Tap to Pay supported
countries one at a time: Canada, UK, Australia, New Zealand,
Singapore, Ireland. EU general + Japan + LatAm wait for v1.3+ or
later — Stripe doesn't support Tap to Pay there yet.

If you're outside the US: sign up for the App Store waitlist anyway
and you'll be among the first when your country rolls out.
```

## CATEGORY 5 — Hostile / trolly

### Q: "This is just a Square clone." / "Stripe could build this."

```
Stripe + Square could absolutely build this. They haven't. The
incentive isn't there: their existing rates pay them more than my
1% would. Open-source-with-AGPL is the explicit hedge against them
building it later — if they fork OSPOS to ship it, AGPL forces
them to share back.

The clone framing isn't wrong. The differentiation is price + license
+ unlimited self-host, not features. I'm fine with that.
```

### Q: "How do I know this won't steal my data?"

```
Fair question. Three answers:

1. Cash transactions never leave your phone. Period. The server
   doesn't have an endpoint to receive them.
2. Source code is published. Anyone can read it. The relevant files
   are src/db/queries.ts (local SQLite layer) and server/src/routes/
   (every endpoint the server exposes). About 4k lines total, an
   afternoon read.
3. Stripe receives card data, not OSPOS. We're a thin client on top
   of Stripe Connect Standard. Stripe's privacy practices govern
   payment data, not ours.

If you find anything that contradicts the above, I want to know.
phil@tttships.co or security disclosures via SECURITY.md.
```

### Q: "You're charging 1% extra on top of Stripe — that's robbery."

```
1% of $30k/year = $300. For comparison: Square charges $1068/year
($89/mo) plus 2.6% + 10¢. Toast charges $828-1980/year ($69-165/mo)
plus 2.49% + 15¢.

For most small merchants the 1% works out to less than they'd pay
for the alternative's monthly fee alone. And there's no monthly fee
to pay if business is slow this month.

If even 1% feels too much, self-host the code. It's AGPL. You set
your own fee (or zero) and your only cost is your server + your
own Stripe account.
```

### Q: "AGPL means this is dead on arrival for serious businesses."

```
AGPL is a legitimate concern for enterprise legal teams. For
small-merchant SaaS that's the customer profile this is targeting,
it's a non-issue — coffee shops and food trucks don't have a legal
team auditing their POS provider's license.

For self-hosting: AGPL only triggers if you MODIFY the source AND
run the modified version as a network service. Running an unmodified
copy for your own business has zero AGPL obligation.

Bitwarden serves 100M+ users under AGPL. The license isn't the
bottleneck people sometimes claim.
```

## CATEGORY 6 — Specific to my situation

### Q: "How long did this take?"

```
6 months of nights and weekends. Started writing the first commit
in March 2026, shipped v1.0 to the App Store May 31, ships v1.1
mid-June, going open source June 30.

I work full-time on something else; OSPOS is the side project I

```

### Q: "Why ship to App Store first instead of open-sourcing day one?"

```
Two reasons:

1. Apple's review cycle is unpredictable. Wanted to clear the
   approval hurdle before flipping the source public — fewer
   moving variables to debug if something rejected.
2. Having a working App Store product makes the OSS launch a
   "here's a thing that actually works and you can install today"
   instead of a "here's some code, maybe build it yourself."

The App Store version is paid hosting on top of the open source
code. Both will always exist.
```

## Don't engage / let it die

Some comments aren't worth responding to:

- Bare snark with no question ("lol nobody will use this")
- Demands for features that contradict the design ("add inventory!")
- Bait questions ("aren't you afraid Stripe will ban you?")
- Anything that turns into a license-war flame ("RMS is wrong about AGPL")

Let those slide. Spend the energy on substantive questions and on
people who sound like they're considering using OSPOS for real.

If a comment is genuinely useful criticism, **don't be defensive.**
Say "you're right, that's a real gap" or "fair, here's the thinking,
let me know if I missed something." Acknowledging valid critique is
a credibility multiplier on HN.
